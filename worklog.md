Task ID: 9-backend
Agent: Backend API Builder
Task: Build Next.js API routes for appointments, contact, newsletter, and LLM chatbot

Work Log:
- Read worklog.md to confirm project context (Islamabad Optical & Clinic, emerald/gold premium site) and prisma schema (Appointment, ContactMessage, NewsletterSubscriber, ChatLog — already pushed to SQLite).
- Loaded the `LLM` skill to learn the exact z-ai-web-dev-sdk API: `await ZAI.create()` -> `zai.chat.completions.create({ messages, thinking: { type: 'disabled' } })`, read `completion.choices[0]?.message?.content`, and crucially the SDK uses the `assistant` role for the system prompt (not `system`).
- Created directories `src/app/api/{appointments,contact,newsletter,chat}/`.
- Wrote `src/app/api/appointments/route.ts` (POST + GET): POST validates required fields (name, email, phone, service, date, time), inserts via `db.appointment.create`, returns `{ success, appointment }` with selected fields; GET uses `db.appointment.groupBy({ by: ['status'] })` to return `{ total, pending, confirmed }`.
- Wrote `src/app/api/contact/route.ts` (POST): validates name/email/subject/message (phone optional), inserts into `ContactMessage`, returns `{ success: true }`.
- Wrote `src/app/api/newsletter/route.ts` (POST): basic email regex validation, normalizes email to lowercase, `findUnique` on email to detect existing subscriber (returns `{ success, alreadySubscribed: true }`) otherwise creates and returns `{ success: true, alreadySubscribed: false }`.
- Wrote `src/app/api/chat/route.ts` (POST): accepts `{ messages, session? }`, sanitizes/validates messages, generates a `crypto.randomUUID()` session if missing, builds SDK messages (prepends the clinic eye-care assistant system prompt as an `assistant` role message per SDK convention, drops client-sent `system` messages), calls ZAI with `thinking: { type: 'disabled' }`, then persists the latest user message + assistant reply into `ChatLog` (best-effort, in a `$transaction`). On any SDK failure or empty reply, returns the graceful fallback "I'm having trouble connecting right now. Please call us at +92 51 111 000 111 or book an appointment online." with status 200. Top-level try/catch also returns the fallback so the UI never crashes.
- Used `NextResponse.json` consistently across all four files; explicit TypeScript interfaces for all payloads; defensive `isNonEmptyString` validation helpers; no `'use client'` (route handlers are server-only by default).
- Ran `bun run lint` — passed cleanly (no errors, no warnings) for all new files.
- Smoke tested live on dev server (port 3000):
  * GET /api/appointments -> 200 `{total, pending, confirmed}`
  * POST /api/appointments (missing fields) -> 400 `{error: "Missing required field(s): ..."}`
  * POST /api/appointments (full body) -> 200 `{success:true, appointment:{id,...,status:"pending",createdAt}}`
  * POST /api/contact -> 200 `{success:true}`
  * POST /api/newsletter (bad email) -> 400 `{error:"Please enter a valid email address."}`
  * POST /api/newsletter (new) -> 200 `{success:true, alreadySubscribed:false}`
  * POST /api/newsletter (duplicate) -> 200 `{success:true, alreadySubscribed:true}`
  * POST /api/chat -> 200 `{reply:"...warm eye-care answer mentioning booking...", session:"<uuid>"}` (real LLM call succeeded, no fallback needed; reply correctly recommends the 20-20-20 rule and asks if user wants to book an exam).
  * After two test appointments, GET /api/appointments -> `{total:2, pending:2, confirmed:0}`.

Stage Summary:
- 4 route files created under `src/app/api/`:
  1. `appointments/route.ts` — POST create + GET stats (groupBy status).
  2. `contact/route.ts` — POST create contact message.
  3. `newsletter/route.ts` — POST upsert subscriber (idempotent, returns `alreadySubscribed`).
  4. `chat/route.ts` — POST LLM eye-care assistant via z-ai-web-dev-sdk with `thinking: { type: 'disabled' }`, session via crypto.randomUUID(), ChatLog persistence, graceful fallback on SDK failure.
- Contracts (request -> response):
  * POST /api/appointments `{name,email,phone,service,date,time,notes?}` -> 200 `{success:true, appointment:{id,name,email,phone,service,date,time,notes,status,createdAt}}` | 400 `{error}` | 500 `{error}`.
  * GET  /api/appointments -> 200 `{total, pending, confirmed}` | 500 `{error}`.
  * POST /api/contact `{name,email,phone?,subject,message}` -> 200 `{success:true}` | 400 `{error}` | 500 `{error}`.
  * POST /api/newsletter `{email}` -> 200 `{success:true, alreadySubscribed:boolean}` | 400 `{error}` | 500 `{error}`.
  * POST /api/chat `{messages:[{role,content}], session?}` -> 200 `{reply:string, session:string}` (fallback reply returned even on internal failure so UI stays alive) | 400 `{error}` (only for empty messages array).
- Notes for frontend agent:
  * All routes accept/return JSON. Use relative paths (`/api/appointments`, etc.) — no port needed.
  * Chat endpoint expects the conversation history in `messages` (role `user`/`assistant`/`system` all accepted; client-sent `system` messages are ignored in favor of the server-side clinic prompt). Persist the returned `session` and pass it back on subsequent turns.
  * Appointments return `status:"pending"` by default; an admin screen (if added later) can flip to `confirmed`/`cancelled`.
  * Lint clean. Dev server healthy.
